## üöÄ Quick Start

### For Web Developers (dormway-lockedin)

1. **Copy types**:
   ```bash
   cd services/dormway-lockedin
   cp ../../shared/types/amplitude-events.ts src/types/
   ```

2. **Create type-safe wrapper**:
   See `AMPLITUDE_PRELAUNCH_IMPLEMENTATION.md` section "Type-Safe Implementation Pattern"

3. **Implement events**:
   Follow day-by-day guide in `AMPLITUDE_PRELAUNCH_IMPLEMENTATION.md`

### For Backend Developers (Engine)

1. **Copy types**:
   ```bash
   cd services/engine
   cp ../../shared/types/amplitude-events.ts src/types/
   ```

2. **Update EventTracker service**:
   See `AMPLITUDE_PRELAUNCH_IMPLEMENTATION.md` section "Backend Implementation"

3. **Update workflows**:
   Replace existing `trackCustomerIOServerEvent` with type-safe `track()`

### For iOS Developers

1. **Install Amplitude SDK**:
   ```swift
   // Add to Package.swift
   .package(url: "https://github.com/amplitude/Amplitude-Swift", from: "1.0.0")
   ```

2. **Convert types to Swift**:
   TypeScript types ‚Üí Swift enums/structs (manual conversion for now)

3. **Implement events**:
   Follow P0 events first: Account, Onboarding, Core Academic

### For Data/Analytics Team

1. **Upload CSV to Amplitude**:
   - Go to Amplitude > Data > Govern
   - Import `amplitude-event-schema-import.csv`
   - This pre-configures all events and properties

2. **Configure CustomerIO Integration**:
   - Amplitude > Destinations > Add Destination ‚Üí Customer.io
   - Map event names (1:1 mapping, same names)

3. **Create Dashboards**:
   - Signup Funnel
   - Onboarding Completion Rate
   - Feature Adoption
   - Error Tracking

## ‚úÖ Best Practices

### Do's ‚úÖ
- Always use TypeScript types for type safety
- Include ALL standard properties on every event
- Use snake_case for all event and property names
- Use past tense for completed actions
- Test events in Amplitude Live View before deploying
- Document any custom events in schema

### Don'ts ‚ùå
- Don't create events without documenting them
- Don't use generic names ("button_clicked")
- Don't skip standard properties
- Don't track PII without consent
- Don't modify event names after implementation
- Don't bypass type-safe wrappers

## üôã FAQ

**Q: Do I need to track ALL 74 events right now?**
A: No! Start with P0 (18 events) before launch, then add P1 (22 events) in Week 1, and P2+ over time.

**Q: Can I create custom events not in the schema?**
A: Yes, but document them in `AMPLITUDE_EVENT_SCHEMA.md` first and ensure they follow naming conventions.

**Q: What if I forget a required property?**
A: TypeScript will catch it at compile time if you use the typed wrapper functions!

**Q: How do I test events before deploying?**
A: Use Amplitude's Live View (real-time debugger) - events appear within 60 seconds.

**Q: What about backward compatibility with existing events?**
A: Since we're pre-launch, we can break existing events and start fresh. No migration needed!

**Q: How do events get to CustomerIO?**
A: Amplitude has a native CustomerIO integration that forwards events automatically.
