# Technical Architecture Patterns

## Purpose
This section contains **implementation architecture patterns** for DormWay platform. These documents answer "HOW do we implement this?" with concrete patterns, flows, and technical details.

This is distinct from [System Architecture](/docs/engineering/architecture/readme), which focuses on architectural decisions and "WHAT we're building" and "WHY."

## Distinction: Technical vs System Architecture

### Technical Architecture (This Section)
- **Focus**: Implementation patterns
- **Audience**: All engineers (backend, full-stack)
- **Questions**: HOW do we implement this?
- **Examples**: "How do we handle retries?", "What's the job processing flow?"

### System Architecture ([System Architecture](/docs/engineering/architecture/readme))
- **Focus**: Decisions and design
- **Audience**: Architects, tech leads
- **Questions**: WHAT are we building? WHY this approach?
- **Examples**: "Why Temporal?", "Why REST over GraphQL?"

## Common Implementation Patterns

### Error Handling Pattern
From Error Handling & Retries:
1. **Classify errors**: Retryable vs non-retryable
2. **Implement retry strategy**: Exponential backoff
3. **Add fallbacks**: Graceful degradation
4. **Log & alert**: Comprehensive observability

### Job Processing Pattern
From Job Processing Flow:
1. **Enqueue job**: Add to Temporal workflow
2. **Process with retries**: Handle transient failures
3. **Update state**: Track progress in database
4. **Emit events**: Publish to event bus (Ably)

### Service Integration Pattern
From External Services Integration:
1. **API client abstraction**: Consistent interface
2. **Circuit breaker**: Prevent cascade failures
3. **Caching layer**: Reduce external calls
4. **Error translation**: Convert external errors to internal types

## Pattern Evolution

### Updating Patterns
When implementation patterns change:
1. Update the document with new approach
2. Document migration path in "Evolution" section
3. Keep old approach documented for reference
4. Update `last_updated` in frontmatter

### Deprecating Patterns
When a pattern is no longer recommended:
1. Mark as `status: deprecated` in frontmatter
2. Add deprecation notice at top
3. Link to replacement pattern
4. Keep for 6 months before archiving
