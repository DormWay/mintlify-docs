## Program Overview

**Duration**: 10 weeks (Nov 18, 2025 - Jan 27, 2025)  
**Cohort Size**: 20 students (manageable for 2 founders)  
**Time Commitment**: 5-8 hours/week  
**Selection**: Rolling applications, first 20 qualified candidates accepted

## Objectives

**For DormWay:**

1. Comprehensive QA across 20+ different campus environments
2. Identify critical bugs and edge cases before January launch
3. Validate AI accuracy (syllabus processing, schedule sync, campus intelligence)
4. Stress test infrastructure under realistic usage patterns
5. Build early evangelists who understand the product deeply

**For Residents:**

1. Learn AI-native product development from founders building in real-time
2. Understand technical decision-making in a fast-moving startup
3. Gain hands-on experience with production SwiftUI, TypeScript, Temporal, and agentic AI
4. Build portfolio evidence of technical contribution
5. Access to professional references and hiring pipeline

## Program Structure

### **Phase 1: Onboarding (Weeks 1-2)**

**Week 1:**

- Kickoff session (90 min): Product vision, architecture overview, current state
- Access provisioning: Staging environment, Slack workspace, Linear read access
- First assignment: Document current environment (campus, LMS, device specs)
- Install DormWay Lock, complete initial setup, report installation issues

**Week 2:**

- Technical deep-dive (90 min): SwiftUI architecture, TypeScript API patterns, Temporal workflows
- Introduction to testing framework and bug reporting standards
- First sprint assignment: Basic functionality testing across 5 core features
- Target: Each resident files 3-5 bugs with detailed reproduction steps

**Deliverables:**

- Environment documentation
- 5+ bugs filed per resident
- Syllabus upload (3 different formats per resident)

### **Phase 2: Active Testing (Weeks 3-8)**

**Weekly Structure:**

- **Monday**: Sprint planning (30 min Zoom) - priority features/areas for testing
- **Wednesday**: Office hours (60 min) - technical questions, bug triage, architecture discussion
- **Friday**: Async update thread - findings, blockers, suggestions
- **Weekend**: Riley streams development work (optional attendance, recorded)

**Testing Focus by Week:**

- **Week 3**: Syllabus processing accuracy (test 50+ syllabi across different formats)
- **Week 4**: LMS sync reliability (Canvas integration, error handling)
- **Week 5**: Schedule intelligence (conflict detection, optimization suggestions)
- **Week 6**: Campus navigation features (location accuracy, building data)
- **Week 7**: AI features (BrainGains, recommendations, notifications)
- **Week 8**: Performance and edge cases (slow networks, large datasets, device compatibility)

**Bi-Weekly All-Hands (60 min):**

- Product roadmap updates and decisions being made
- Demo of new features built that week
- Discussion of most interesting bugs/insights
- Q&A on startup operations, fundraising, technical architecture

**Deliverables:**

- 10+ bugs per resident per week
- 2-3 feature suggestions with rationale
- Weekly testing report (template provided)

### **Phase 3: Pre-Launch Validation (Weeks 9-10)**

**Week 9: Stress Testing**

- Coordinated load testing: All 20 residents active simultaneously
- Regression testing on all previously filed bugs
- Documentation review: Does onboarding make sense to new users?
- Final feature requests and prioritization

**Week 10: Launch Preparation**

- Final bug sweep and critical issue verification
- Success story documentation: What changed based on resident feedback?
- Resident showcase: Present most impactful contributions
- Transition planning: Continue as alpha users or move to recruiting pipeline

**Deliverables:**

- Comprehensive test coverage report
- Launch readiness assessment per resident
- Personal case study of contribution impact

## Risk Mitigation

### **Potential Issues**

**1. Low Application Quality**

- Mitigation: Lower bar to 15 residents if necessary, focus on diversity of campuses
- Backup: Direct recruit from Riley's Twitch community and known networks

**2. Poor Engagement/Dropout**

- Mitigation: Weekly check-ins with less active residents, understand blockers
- Backup: Accept rolling applications to backfill if someone drops

**3. Overwhelming Bug Volume**

- Mitigation: Clear bug severity guidelines, automated duplicate detection
- Strategy: Weekly triage sessions, prioritize critical/high severity

**4. Infrastructure Not Ready**

- Mitigation: Staged rollout - start with 5 residents Week 1, expand if stable
- Backup: Delay program start by 1 week if critical issues found

**5. Founder Time Constraints**

- Mitigation: Pre-record some content, make more async
- Backup: Scale back office hours frequency if necessary
